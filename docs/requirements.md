

# 法院专项账务记账系统业务需求说明书 (V2.0 优化版)

## 一、 系统定位与核心架构

### 1.1 系统名称
法院专项账务记账系统 (Court Special Accounting System, CSAS)

### 1.2 系统定位
本系统是专为法院内网设计的**通用型记账平台**。采用“微内核+业务插件”的设计思想，底层为标准的双向分录记账引擎，上层通过配置模板适配案款、诉讼费、食堂、工会等多种专项账务。

### 1.3 核心设计原则
*   **多维隔离：** 支持多账套、多年度物理/逻辑隔离。
*   **通用引擎：** 凭证引擎不关联具体业务字段，所有业务属性通过“辅助核算”动态扩展。
*   **业财融合（优化）：** 支持从外部系统（一张网）导入收退费流水并按规则自动生成凭证。
*   **核销管理（优化）：** 针对案款业务，引入“往来核销”机制，追踪每一笔资金的流入与流出对应关系。

---

## 二、 系统边界

### 2.1 负责内容
*   **全流程账务处理：** 凭证录入、审核、结账、打印、核销、报表生成。
*   **辅助核算管理：** 案号、收据号、当事人、供应商等维度的明细核算。
*   **出纳与对账：** 银行流水导入、手动/自动对账、银行存款余额调节表。

### 2.2 不负责内容
*   不干涉外部系统的业务审批流（如退费审批）。
*   不直接处理物理现金/转账动作（仅记录账务结果）。

## 三、 核心数据模型 (Database Schema)

### 3.1 核心表结构说明

#### 3.1.1 凭证主表 (fin_voucher,按年分表如 fin_voucher_2026)
| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| voucher_id | UUID (varchar 36) | 主键 |
| account_set_id | UUID (varchar 36) | 账套ID(单院部署,仅做账套隔离) |
| period | String | 会计期间 (如 2026-05) |
| voucher_no | Int | 凭证号 (账套+期间内唯一) |
| status | Enum | DRAFT 草稿 / SUBMITTED 未审核 / AUDITED 已审核 / PRINTED 已打印 / VOIDED 已作废 |

#### 3.1.2 凭证明细表 (fin_voucher_detail,按年分表)
| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| detail_id | UUID (varchar 36) | 主键 |
| subject_code | String | 会计科目编码 |
| debit_amount | Decimal(18,2) | 借方金额 |
| credit_amount | Decimal(18,2) | 贷方金额 |
| verification_status | String | **核销优化：** NOT_REQUIRED 无需核销 / UNVERIFIED 未核销 / PARTIALLY_VERIFIED 部分核销 / FULLY_VERIFIED 已核销 |
| aux_desc | String | **性能优化：** 冗余存储所有辅助核算信息的文本摘要(`code:label;` 拼接) |

#### 3.1.3 凭证辅助核算值表 (fin_voucher_aux_value)
| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| detail_id | UUID | 关联明细 |
| aux_type_code | String | 辅助项类型 (case_no, supplier_id等) |
| aux_value | String | 实际值 (对应档案ID或手动输入文本) |

## 四、 功能模块详述

### 4.1 智能凭证中心
*   **多模式录入：** 支持手工录入、Excel批量导入、以及“业务流水转凭证”。
*   **动态辅助核算：** 选择科目后，根据配置（`fin_subject_aux_config`）动态渲染输入框（如选择“案款”科目，自动弹出案号、当事人必填项）。
*   **智能断号处理：** 提供结账前一键整理断号功能，支持按日期重排凭证号。

### 4.2 往来核销管理 (核心优化)
针对案款、往来款科目，提供“逐笔核销”功能：
*   **核销入口：** 用户可选择某一案号下的“借方”和“贷方”明细进行对冲。
*   **自动勾对：** 系统可根据“案号+金额”或“收据号”自动匹配核销记录。
*   **核销报表：** 生成《账龄分析表》及《未核销案款明细表》，监控超期未清退款项。
*   在处理 `debit` 或 `credit` 时，需根据科目配置判断是否触发核销状态变更。
* “针对‘案款’、‘往来’类科目，必须支持按**辅助核算项（案号+收据号）**进行往来核销，并能据此生成‘未核销明细报表’。”

### 4.3 账簿与跨账套查询 (报表优化)
*   **标准账簿：** 总账、明细账、科目余额表、科目汇总表、辅助核算余额表。
*   **汇总工作台：** 支持勾选多个账套（如案款账+诉讼费账），一键生成汇总的科目余额，方便领导查看全院专项资金分布。


### 4.4 出纳管理 (新增)
*   **银行流水导入：** 支持主流银行Excel/CSV流水导入。
*   **对账引擎：**
    *   **自动对账：** 按日期、金额、收据号自动关联凭证。
    *   **对账单：** 自动生成《银行存款余额调节表》，标注“企业已付银行未付”等差异项。

### 4.5 期末处理
*   **自动结转：** 支持期末损益自动结转至净资产/本期盈余。
*   **严格结账检查：** 
    *   检查所有凭证是否已审核。
    *   检查辅助核算项是否漏填。
    *   检查往来科目是否存在余额为负的异常情况。
*   **反结账内控：** 反结账需记录操作理由，并抄送审计日志。

## 五、 系统非功能性要求

### 5.1 安全与内控 (审计优化)
*   **审计员角色：** 独立于系统管理员，仅拥有只读权限和全量日志审计权限。
*   **国产化适配：** 服务端需支持银河麒麟/统信UOS，数据库适配达梦、人大金仓。

### 5.2 性能指标
*   **并发支持：** 满足全院出纳、会计同时在线操作（支持不少于 50 个并发连接）。
*   **查询性能：** 跨年度、跨账套明细账查询响应时间 $\le 3s$。
*   **高可靠性：** 支持定时自动备份（全库备份+账套独立备份）。

## 六、 推荐算法与逻辑公式

### 6.1 科目余额计算逻辑
对于任意科目 $i$ 在期间 $t$ 的期末余额 $E_i^t$：
$$E_i^t = S_i^t + (D_i^t - C_i^t) \times Direction$$
其中：
*   $S_i^t$: 期初余额
*   $D_i^t$: 本期借方发生额合计
*   $C_i^t$: 本期贷方发生额合计
*   $Direction$: 余额方向（借方为 1，贷方为 -1）

### 6.2 核销状态转换逻辑
*   `unverified`: $\sum |核销金额| = 0$
*   `partially_verified`: $0 < \sum |核销金额| < |明细原额|$
*   `fully_verified`: $\sum |核销金额| = |明细原额|$

## 技术栈：政法政务通用版

### 1. 后端 (Backend): PHP / ThinkPHP 生态
*   **核心框架：** `PHP 7.4.33` + `ThinkPHP 5.0.24` 风格（单入口、`action` 路由分发）。
    *   **理由：** 法院内网普遍部署 MAMP/Nginx + PHP-FPM 等轻量环境，部署与运维成本低；与院内已有 PHP 业务系统（如 cccf）架构一致，可复用既有运维与备份体系。
*   **持久层：** ThinkPHP `Db` 查询构造器(原生 SQL/Query Builder 双轨)。
    *   **理由：** 财务系统涉及大量复杂查询和手动 SQL 优化。`Db` 类语法贴近 SQL，方便在处理“多账套”、“跨年度汇总”时进行 SQL 调优；其驱动层抽象也便于后续平滑迁移到达梦/人大金仓。
*   **权限安全：** 自定义中间件 + `JWT`(`firebase/php-jwt` 或网关签发)，请求层注入 `X-User-Id` / `X-User-Name` 上下文。
*   **报表导出：** `PhpSpreadsheet`(Excel 导入/导出，凭证批量、银行流水)+ 前端 `vue-print-nb` / `Print.js`(凭证套打)。
    *   **理由：** PhpSpreadsheet 提供 `ReadFilter` + 行游标 API,可流式处理上万行流水,避免内存峰值;凭证打印走前端模板更便于按法院定制版式。

### 2. 前端 (Frontend): Vue 3 体系
*   **核心框架：** `Vue 3 (Composition API)` + `Vite`。
*   **UI 组件库：** `Element Plus`。
    *   **理由：** 国内政务系统 UI 规范的标配，文档齐全，对财务大表单支持极好。
*   **状态管理：** `Pinia` (替代 Vuex，更轻量)。
*   **打印组件：** `vue-print-nb` 或 `Print.js`。

### 3. 数据库与存储 (Database & Storage)
*   **关系型数据库：** `MySQL 8.0`(开发环境);生产环境按法院信创要求适配 `达梦 DM8` / `人大金仓 KingbaseES`。
    *   **理由：** ThinkPHP `Db` 抽象层 + 严格使用 ANSI SQL/Query Builder,可在 MySQL 与国产库之间平滑迁移,避免方言锁定。
*   **凭证类大表分年存放:** `fin_voucher_YYYY` / `fin_voucher_detail_YYYY` / `fin_voucher_aux_value_YYYY`,降低单表体量,便于跨年归档与备份。
*   **附件存储:** 本地文件系统(`backend/public/uploads/<account_set>/<yyyy-mm>/`),按账套+月份分目录;后续可对接院内文件服务器或对象存储。
